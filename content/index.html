FaaSificação de código. Entenda essa abordagem.

Simplicidade, baixas barreiras de entrada e um caminho rápido para o primeiro sucesso estão entre os principais requisitos para garantir que os conceitos e tecnologias sejam escolhidos pelos engenheiros de software. Um dos paradigmas de entrega de software mais promissores é a computação sem servidor, que gerou dezenas de novas tecnologias de tempo de execução e ofertas comerciais. Os métodos e ferramentas de desenvolvimento de aplicativos correspondentes levaram mais tempo para aparecer.

FaaSificação de código. Entenda essa abordagem.

Aplicativos de engenharia de software e implantação de serviços baseados em software em nuvem são comparáveis ​​a distribuir convites para um jantar para o mundo inteiro. Não sabemos quem chegará, como se comportarão, se podemos acomodar a todos e se o resultado será positivo afinal. Em termos técnicos, o software precisa estar preparado para lidar com milhares de solicitações pesadas paralelas sob demanda, sem alocar recursos caros quando apenas alguns usuários (ocasionalmente) causam uma carga leve. As nuvens de hoje oferecem suporte a três estratégias principais para atingir esse objetivo.

O primeiro é o uso de autoescaladores reativos e escalonadores manuais baseados em previsões proativas para ajustar a alocação de recursos em tempo hábil para a carga atual. A segunda é simplesmente provisionar um serviço PaaS e deixar que o provedor se preocupe com a estratégia de escalonamento. O último é o uso de Function-as-a-Service (FaaS) que oculta essa complexidade e executa tantas instâncias quantas forem as solicitações, ao custo de uma pequena sobrecarga de inicialização, dependendo do isolamento subjacente tecnologia. Essa ocultação da complexidade pode ser estendida a outros aspectos da hospedagem do aplicativo, levando a uma mentalidade de fato de descuido (no positivo sentido) computação sem servidor.

Essa mentalidade leva um tempo para se adaptar, mas sua atratividade intrínseca para engenheiros de software está na evolução de DevOps e GitOps para NoOps, onde muitos questões convencionais são erradicadas. Como as instâncias de função geradas em FaaS são de curta duração e de-facto (embora não de jure) sem estado, as invocações tornam-se mais previsíveis porque vazamentos, processos suspensos, esgotamento de recursos ou outros problemas que geralmente aparecem gradualmente ao longo do tempo são evitados por design.

A etapa crucial é tornar um aplicativo de software sem servidor em primeiro lugar. De uma perspectiva de design (acoplando microsserviços sem estado de curta duração) para uma perspectiva de ferramentas (descrevendo e implantando a composição), o FaaSification auxilia os engenheiros a realizar esta etapa com sucesso. FaaSification é o processo de conversão de uma estrutura de código em um formato executável em FaaS.

Os sistemas de TI de hoje são grandes, dinâmicos, complexos e heterogêneos. Ao adotar uma cultura DevOps, os desenvolvedores devem adotar uma abordagem de desenvolvedor full stack, na qual assumem a responsabilidade pelo ambiente de teste e lançamento. Eles devem dominar um conjunto de habilidades estendidas além do conhecimento do código. Toda a experiência acumulada em infraestrutura pode ser escrita em artefatos para compor um processo de implantação do GitOps. Nesse contexto, se os procedimentos de manutenção forem adicionados, como a detecção de anomalias, um ambiente NoOps ocorrerá.

Os aplicativos FaaS estão um passo à frente. Nessa abordagem, os desenvolvedores ficam isentos da responsabilidade de lidar com os aspectos da infraestrutura, incluindo o escalonamento, uma vez que o provedor deve garantir que o código implantado seja processado em uma solicitação separada até que todos os recursos estabelecidos no contrato com o provedor sejam consumidos. Normalmente, ele permite que um aplicativo responda em diferentes cenários de carga de trabalho, sem nenhuma interação. Por outro lado, os desenvolvedores têm um aspecto novo e às vezes bastante complexo com o qual se preocupar: a pilha do provedor. Para piorar a situação, vem ganhando espaço um paradigma multicloud, no qual muitos provedores oferecem serviços semelhantes, com comportamentos distintos.

Porém, em ambientes orientados a FaaS, embora a tarefa de gerenciar a infraestrutura seja suprimida, novos requisitos são adicionados à lista de conhecimentos que o desenvolvedor precisa dominar. Afinal, publicar o código desenvolvido exige que o profissional saiba como funciona o serviço oferecido pelo provedor, como funciona o console e ainda tenha intimidade com a API, já que provavelmente precisará automatizar as tarefas de implantação.

Para tornar a vida dos desenvolvedores menos dolorosa e ajudá-los a se livrar das preocupações com a infraestrutura de seus aplicativos, surgiram abordagens que ajudam no processo de implantação de aplicativos.

Algumas ferramentas podem até ser usadas para implementar aplicativos orientados a FaaS, mas não foram especificamente projetadas para isso. Terraform, CloudFormation e Cloudify são exemplos desse tipo de abordagem. Terraform é uma ferramenta de software de infraestrutura como código de código aberto criada pela HashiCorp. Ele permite que os usuários definam e provisionem uma infraestrutura de datacenter usando uma linguagem de configuração de alto nível conhecida como Hashicorp Configuration Language (HCL) ou, opcionalmente, JSON. O CloudFormation torna possível usar linguagens de programação ou um arquivo de texto simples para modelar e provisionar automaticamente e com segurança todos os recursos necessários para aplicativos em todas as regiões e contas na AWS.

Existe uma classe de ferramentas que são projetadas especificamente para auxiliar na adoção da abordagem FaaS.

Claudia é uma biblioteca Node.js que facilita a implantação de projetos Node.js para AWS Lambda e API Gateway. Ele automatiza todas as tarefas de implantação e configuração sujeitas a erros e configura tudo da maneira que os desenvolvedores de JavaScript esperam fora da caixa. Claudia é construída em cima do AWS SDK para tornar o desenvolvimento mais fácil. Não é um substituto para o AWS SDK ou AWS CLI, mas uma extensão que torna algumas tarefas comuns, como implantação e configuração de gatilhos, fáceis e rápidas. Claudia atua como uma ferramenta de linha de comando, e permite criar e atualizar funções a partir do terminal. O ecossistema Claudia vem com duas outras bibliotecas Node.js úteis: Claudia API Builder permite que você crie a API no API Gateway, e Claudia Bot Builder permite criar chatbots para várias plataformas de mensagens.

Zappa é uma ferramenta de linha de comando que Lambda-fies Flask e aplicativos Django. É uma estrutura que empacota e implanta aplicativos Python compatíveis com WSGI para uma função AWS Lambda e AWS API Gateway. Zappa é um botão fácil para a implantação sem servidor Flask e Django e também permite construir aplicativos híbridos orientados a eventos que podem escalar para trilhões de eventos por ano sem nenhum esforço adicional.

PyWren expõe um primitivo de mapa perfeito do Python no topo do AWS Lambda. Embora o Lambda tenha sido projetado para executar microsserviços baseados em eventos em escala, ao extrair dinamicamente o código do S3 ele faz com que cada invocação do Lambda execute uma função diferente. PyWren serializa uma função Python, capturando todas as informações relevantes, bem como a maioria dos módulos que não estão presentes no tempo de execução do servidor. Isso elimina a maior parte da sobrecarga do usuário sobre implantação, empacotamento e controle de versão de código. Essa ferramenta envia a função serializada junto com cada datum serializado, colocando-os em chaves globalmente exclusivas no S3 e, em seguida, chama uma função Lambda comum. No lado do servidor, ele invoca a função relevante no datum relevante, ambos extraídos de S3. O resultado da chamada da função é serializado e colocado de volta no S3 em uma chave pré-especificada, e a conclusão do trabalho é sinalizada pela existência dessa chave.

Lambada é uma ferramenta não intrusiva que funciona com o código Python vanilla existente e, portanto, tem como alvo engenheiros de software e desenvolvedores sem qualificações especializadas em computação em nuvem. O Lambada importa dinamicamente o código do aplicativo, inspeciona o namespace e transforma funções e classes em proxies locais emparelhados com funções remotas geradas. O objetivo principal é executar o código Python através do Lambada para executá-lo no ambiente Lambda escalável, resiliente e centrado na invocação.

Podilizer é implementado em Java, usando a estrutura JavaParser para análise de código estático e a biblioteca Maven para injetar instruções de construção nos projetos de software de destino. Ele executa a ferramenta AWS CLI como processo externo para todas as interações com o AWS Lambda e qualquer estrutura de teste de unidade compatível que encontrar para a etapa de verificação.

O framework Node2FaaS é uma ferramenta para conversão automática de aplicativos monolíticos, escritos em Node.js, para aplicativos orientados a FaaS. Essa ferramenta lê o código original do aplicativo de destino e o converte em um aplicativo cujas funções são executadas em um serviço FaaS. O código interno das funções é convertido em implantações executadas em funções criadas automaticamente no provedor. A solicitação de serviço é colocada no lugar da chamada de função.

Function as a Service: o que é e para quê serve?

Recentemente os provedores passaram a oferecer um modelo de serviço que entrega uma plataforma completamente encapsulada para desenvolvimento de software e cobra apenas pelo processamento efetivamente executado pela plataforma. Esse tipo de serviço, conhecido como Function as a Service (FaaS), tem ganhado bastante evidência, uma vez que a sua abordagem vai ao encontro do modelo arquitetural que tem sido bastante difundido na atualidade, que é o paradigma de microsserviços.

Function as a Service: o que é e para quê serve?

Microsserviços

O termo microsserviços tem sido utilizado desde 2014 em comunidades de desenvolvimento ágil. Os padrões e os princípios que permeiam o conceito de microsserviços incluem: Desenvolvimento orientado ao negócio e nativamente baseado em nuvem; Aplicação de múltiplos paradigmas de desenvolvimento, como funcional e imperativo; Aplicações executando em serviços leves de contêiner, como Docker; Entrega contínua descentralizada; Utilização de cultura DevOps (desenvolvimento integrado à operação); Utilização de API REST.

Para a construção de software, por meio dos princípios de microsserviços, é necessário segmentar o desenvolvimento das funcionalidades da aplicação. A Figura acima mostra a comparação da abordagem de microsserviços com a abordagem monolítica. Como pode ser visto, na abordagem monolítica toda a aplicação é colocada dentro de um único processo, enquanto que na abordagem de microsserviços, cada funcionalidade é colocada em um serviço diferente. Com isso, na abordagem monolítica toda a aplicação precisa ser escalada, no caso de microsserviços, apenas aqueles serviços com maior demanda serão escalados (proporcionando maior racionalidade no consumo de recursos)

É nesse contexto que os serviços de Function as a Service entram!

Nele o cliente contrata a execução de uma função predefinida, carrega o código que deseja executar, e recebe um endereço de acesso ao serviço. As aplicações que utilizam esse tipo de serviço de nuvem têm sido denominadas aplicações serverless, uma vez que a aplicação não possui um servidor específico e seu funcionamento é baseado em requisições feitas à API do provedor.

Os serviços de FaaS oferecem como principais benefícios da sua adoção: Supressão da necessidade de gerenciamento de servidores; Elasticidade contínua automática; Pagamento sob demanda; Aumento da flexibilidade do projeto e Redução dos riscos.

Atualmente, os principais provedores de nuvem pública possuem serviços de FaaS em seus catálogos. A pioneira Amazon, oferece o serviço AWS Lambda. Esse serviço permite processamento de NodeJS, Python, Java, Go, Ruby e C\# através do .NET Core e disponibiliza um pacote mensal de um milhão de requisições gratuitas, antes de iniciar a cobrança pelo serviço. O provedor possui diversas parcerias para a implantação, o monitoramento, o gerenciamento de código e a segurança. O Google oferece o serviço Cloud Functions. Esse serviço permite processamento de funções escritas em Python, NodeJS e Go, e disponibiliza uma cota inicial de dois milhões de requisições por mês e inicia a cobrança apenas quando essa cota é ultrapassada. A Microsoft, por meio do seu serviço de nuvem Azure, disponibiliza o serviço Azure Functions. Esse serviço permite o carregamento de funções escritas em C#, F#, NodeJS, Java, Python ou PHP.

Computação em Nuvem

Há mais de uma década, quando a Amazon, gigante americana do varejo, resolveu se aventurar no desconhecido mundo dos provedores de nuvem, não era possível imaginar quão importante se tornaria o novo conceito que a empresa estava lançando.

No meio acadêmico, o conceito de nuvem não era exatamente uma novidade, um vez que os GRIDs computacionais já eram utilizados há bastante tempo por grandes universidades que compartilhavam recursos computacionais através da internet. No entanto, imaginar que bastaria um cartão de crédito e alguns minutos para ter acesso a recursos computacionais quase ilimitados era apenas um sonho.

Fato é que o sonho tornou-se realidade!

A computação em nuvem é tão presente que sequer a percebemos mais! Hoje em dia, empresas iniciantes geralmente suprimem das suas planilhas de abertura custos com implantação de Datacenter. Mesmo sabendo que em algum momento vão precisar de tecnologia computacional, não se preocupam com isso, pois sabem que é tão fácil quanto abrir uma conta de e-mail.

Embora em 2011 o National Institute of Standards and Technology, NIST, tenha normatizado alguns conceitos que classificam a Computação em Nuvem, hoje em dia é possível encontrar modelos de serviço bem diversificados. Além de infraestrutura como serviço (IaaS), plataforma como serviço (PaaS) e software como serviço (SaaS), então  padronizados pelo NIST, hoje é possível encontrar provedores oferecendo diversos modelos de serviços de nuvem, como: banco de dados, map reduce, log, monitoramento, funções, container, dentre muitos outros. Todos são fornecidos como serviço, sem qualquer interação do cliente com a infraestrutura. Essa profusão de modelos tem sido denominada XaaS (qualquer coisa como serviço).

Computação em Névoa

Grande parte do tráfego da internet atualmente está associada a serviços em nuvem. Por conta disso, a disputa pela rede se tornou um problema cada vez mais desafiador. Uma forma de atenuar os problemas causados por essa concorrência é trazer o recurso computacional mais pra perto, ou seja, em vez de sempre ir até o servidor para utilizar um recurso, algum dispositivo no meio do caminho poderia fazer as vezes do servidor. Diversos dispositivos já estão habilitados pra fazer isso, tais como: smart switches, smart hubs ou smart roteadores. Eles podem estar em algum provedor no meio do caminho entre a máquina do cliente e o Datacenter do provedor. Quanto mais perto, melhor.

Esse modelo tem várias denominações, dentre elas computação em névoa, neblina, fumaça e nevoeiro, todas derivadas do termo “Fog Computing”. Embora esse modelo traga ganhos em termos de desempenho tanto de processamento quanto de transmissão de dados, sua adoção enfrenta diversos problemas de segurança e inconsistência de dados. Além disso, como seu funcionamento depende de investimentos em equipamentos físicos, sua implementação também envolve um custo elevado.

Computação no Orvalho

Paralelamente ao crescimento da computação em nuvem, também cresceu significativamente o número e os tipos de dispositivos pequenos conectados à internet. Além dos telefones celulares, câmeras de segurança, eletrodomésticos, veículos, drones, set-top boxes e muitos outros dispositivos passaram a contar com conexão com a internet.

Inicialmente, esses dispositivos pequenos eram enxergados apenas como clientes. Porém, tanto poder computacional geograficamente distribuído despertou na comunidade a possibilidade de usá-los também como provedores de recursos, seja para armazenamento ou processamento. Imagine se sua geladeira fizesse um cache daquela página de culinária que sua vizinha tanto acessa e, ao invés dela precisar viajar por toda a internet até o servidor, a rede fosse liberada pra você assistir seus filmes no streaming? Maravilha, não? Se a sua geladeira ainda não está fazendo isso, talvez seu celular trabalhe assim com o seu smart watch. Esse modelo de computação tem sido denominado Computação no Orvalho (Dew Computing).

Computação na Selva

A computação de alto desempenho, muito utilizada por pesquisadores mundo afora (seja para descoberta do DNA ou para qualquer outra grande contribuição científica), demanda muito recurso computacional. Atualmente, existe um enorme poder computacional espalhado pelo mundo, inclusive um pequeno pedaço na palma da sua mão.

A computação na selva é um paradigma de computação de alto desempenho, no qual um sistema distribuído congrega recursos computacionais presentes em diversas fontes, desde nuvens, GRIDs, supercomputadores, clusters e até pequenas máquinas, como computadores pessoais e dispositivos móveis.
 

Computação na poeira

Poeira inteligente (SmartDust) são computadores super pequenos (do tamanho de um grão de arroz ou até menores), que trabalham em conjunto formando uma rede de sensores sem fio. Com esses dispositivos, seria possível lançar milhares de pequenos dispositivos em um local e então formar uma rede ad-hoc a partir dos sensores. Essa rede poderia ser utilizada para diversas atividades, em especial aquelas com maiores níveis de sensibilidade, como por exemplo detecção de substâncias químicas no ambiente.
Desde meados de 1990, esse conceito vem sendo estudado. Atualmente, já existem microcâmeras do tamanho de um grão de sal, que podem ser inseridas no corpo humano e auxiliar em pesquisas a nível celular.

Computação no céu

Considerando o grande número de provedores oferecendo serviços de nuvem atualmente, por que ao invés de ter um provedor não temos vários? O conceito de sistemas multi cloud tem crescido significativamente, e com ele várias ferramentas para promover a administração e orquestração dessas nuvens.
 
Os benefícios da utilização dessa abordagem vão desde alta disponibilidade, melhor desempenho, maior distribuição geográfica, entre outros. Para empreendimentos globais, essa abordagem é quase obrigatória, afinal não é razoável fazer um cliente na Ásia atravessar o Pacífico para acessar uma informação que poderia estar hospedada em um provedor a alguns quilômetros da sua casa.
Para abstrair a complexidade das APIs de cada provedor, a definição de infraestrutura como software (Infra as code) se tornou fundamental para o resultado da implementação do modelo de computação no céu. A diferença entre o sucesso e o fracasso de um projeto pode estar na (in)capacidade de promover a correta abstração dessas integrações.
